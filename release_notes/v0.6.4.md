# PyWiki v0.6.4 — Release Notes

**Released:** 2026-03-06
**Branch:** `devel`
**Tag:** `v0.6.4`

---

## Highlights

Search overhaul — filters by format, author, date range, and category;
plus a fix for substring matching on PostgreSQL.

---

## Added

### Search filters

The `/search` page now has a collapsible **Filters** panel with:

- **Namespace** — restrict to a single namespace
- **Format** — Markdown / RST / Wikitext
- **Author** — filter by the username of the last editor
- **Updated from / to** — ISO date range pickers

The filter panel opens automatically when any filter is active.
A **Clear filters** link appears when filters are set.

### `Category:Name` search syntax

Type `Category:Python` in the search box to find all pages tagged with
that category. Matches both `[[Category:Python]]` (Markdown/wikitext)
and `.. category:: Python` (RST).

### Filter-only queries

The search box can now be left empty when filters are active — useful
for e.g. "show me all Markdown pages by author plh":

```
/search?format=markdown&author=plh
```

`q=*` is also treated as match-all.

### Result improvements

Each search result now shows a format badge and the author's username
(linked to their profile).

---

## Fixed

### Substring matching on PostgreSQL

Previously, PostgreSQL used `plainto_tsquery` (word-token FTS) as the
match filter, so partial terms like `mage` would not find pages
containing `image`. Search now uses ILIKE `%term%` for matching on both
SQLite and PostgreSQL. On PostgreSQL, FTS is still used to **rank**
results (most relevant first), but ILIKE determines which pages are
included.

### Filter-only search returning no results

When no search text was given but filters were active, the PostgreSQL
path would fall into the FTS branch and return nothing. Filter-only
queries now correctly use the ILIKE/no-text path.

## Upgrade Instructions

```bash
cd /opt/pywiki
git pull origin main
systemctl stop pywiki && systemctl start pywiki
```

No database migrations required for this release.
